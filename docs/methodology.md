# Methodology

## Overview

We investigate whether retrieval-augmented generation (RAG) improves LLM-based
detection and classification of bugs in Qiskit quantum programs compared to
prompt-only and rule-based baselines.

## Research Questions

1. Does augmenting an LLM prompt with retrieved bug-pattern context (RAG)
   improve bug classification accuracy on Bugs4Q relative to the prompt-only
   baseline?
2. Does the lightweight rule-based static baseline provide a competitive
   lower-bound for precision on common, syntactically visible bug patterns?
3. Which taxonomy classes benefit most from retrieved context?

## Datasets

### Bugs4Q (Benchmark / Evaluation)

Bugs4Q (<https://github.com/Z-928/Bugs4Q>) is an executable benchmark of
real Qiskit bugs, each with a buggy version, a fixed version, and a
ground-truth classification label.  We use Bugs4Q as the **sole source of
evaluation labels** for all reported metrics.

### Bugs-QCP (Knowledge-Base Corpus)

Bugs-QCP (Zenodo 5834281) is a broader corpus of quantum computing program
bugs.  We use it exclusively for **knowledge-base enrichment and taxonomy
grounding** in the RAG pipeline.  Bugs-QCP samples are never used as
evaluation targets.

## Experimental Modes

| Mode | Description |
|------|-------------|
| `static` | Rule-based heuristic matching over source text. No LLM. Lightweight placeholder only. |
| `prompt_only` | LLM inference on the raw code snippet with no additional context. |
| `rag` | LLM inference with the top-*k* most similar bug patterns from the knowledge base prepended to the prompt. |

## Retrieval

We use a TF-IDF + cosine-similarity retriever over the knowledge-base pattern
descriptions and example code.  The default value of *k* is 5 (configurable
in `config.yaml`).  Dense retrieval (Sentence-Transformers + FAISS) is the
intended next step and can be plugged in via `src/retriever.py`.

## Evaluation Metrics

We report the following classification metrics on Bugs4Q, computed using
scikit-learn:

- **Accuracy** — fraction of samples correctly classified.
- **F1 (macro)** — macro-averaged F1 across all taxonomy classes.
- **Precision (macro)** — macro-averaged precision.
- **Recall (macro)** — macro-averaged recall.
- **Per-class F1** — F1 per taxonomy class for error analysis.

## Leakage Control and Split Discipline

To prevent knowledge-base leakage into evaluation:

1. The knowledge base is constructed from the **Bugs-QCP corpus** and,
   optionally, the **Bugs4Q training split** only.
2. The **Bugs4Q evaluation split** is never used to populate the knowledge
   base or retrieved context.
3. Retrieved patterns in RAG mode must not expose evaluation-set ground-truth
   labels to the model.
4. We document the split configuration (train vs. eval indices) alongside each
   published result.

## Smoke-Test vs. Benchmark Runs

Synthetic data generated by `prepare_bugs4q.py --smoke-test` is for verifying
that the pipeline infrastructure executes without errors.  It is never reported
as a benchmark result.  All reported metrics are produced from Bugs4Q data
only.

## Limitations

- The static baseline uses only textual pattern matching; it is not a faithful
  re-implementation of any published quantum static analyser such as LintQ.
- The RAG retriever currently uses TF-IDF, which may miss semantically similar
  patterns that share few surface tokens.
- Evaluation on the full Bugs4Q benchmark is in progress; results reported at
  the scaffold stage are preliminary.
